The Problem

github is not a backup strategy.

Accidental force-pushes, compromised accounts, repo deletions, API outages — your code lives in one place and you don't control the infrastructure. If GitHub goes down or a repo disappears, what's your recovery plan?

Most backup solutions are either manual scripts that rot, paid services with opaque pricing, or enterprise tools that are overkill for small teams. You need something that runs itself, costs almost nothing, and stores backups on infrastructure you own.

In Practice

set up once, never think about it again.

register a repo

Point gitcask at any GitHub repository. Set your backup interval in minutes. The scheduler handles the rest — no cron jobs to maintain, no scripts to babysit. 

# add a repo with hourly backups
$ curl -X POST https://your-worker.dev/repos \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"owner":"org","name":"app","interval_minutes":60}'

trigger on demand

About to do something risky? Trigger an immediate backup before that force-push, migration, or branch cleanup. One curl and you've got a snapshot. 

# trigger an immediate backup
$ curl -X POST https://your-worker.dev/repos/<id>/trigger \
  -H "Authorization: Bearer $TOKEN"

# check the run status
$ curl https://your-worker.dev/repos/<id>/runs \
  -H "Authorization: Bearer $TOKEN"

retention policies

Configure how many backups to keep. Old archives are automatically cleaned up on every scheduler tick — your R2 bill stays predictable without manual pruning. 

# keep the last 30 backups per repo
$ curl -X PATCH https://your-worker.dev/repos/<id> \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"retention_count": 30}'

Why Gitcask

no agents, no dashboards, no invoices.

  • vs. manual scripts — No cron jobs to maintain on a VPS that may itself go down. Gitcask is serverless and self-healing.
  • vs. GitHub Actions — Actions can't run if GitHub is the thing that's broken. Your backups should be independent of what they're backing up.
  • vs. paid backup SaaS — No per-repo pricing, no seat licenses. Deploy to your own Cloudflare account and pay only for storage.
  • vs. enterprise tools — No SAML setup, no sales calls, no 50-page admin console. One Worker, one config, done.

The Stack

cloudflare all the way down.

Workers for the API and scheduler. D1 for repo metadata and run history. R2 for storing compressed mirror archives. Queues for reliable job delivery. Containers for running git clone.

Everything deploys with a single wrangler deploy. No Docker compose files, no Kubernetes manifests, no infrastructure to monitor. The entire system fits in one repo.

start backing up today.
