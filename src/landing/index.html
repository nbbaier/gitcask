Why it exists

Own the backup path.

GitCask is for teams that want GitHub repository backups in their own Cloudflare account, under their own retention rules.

Mirror, don’t approximate.

Backups use mirrored Git clones so branches, tags, and refs are captured from the source instead of reconstructed later.

Keep the surface operational.

Repositories, schedules, manual triggers, run history, and job status stay scriptable through the API and CLI.

Workflow

Short path from repo to archive.

The system stays narrow on purpose. Register a repository, set an interval, let the scheduler dispatch work, and inspect the resulting artifacts when a run completes. No broad platform story. Just a reliable backup loop.

01

Register the repository

Add the owner, name, and backup interval through the API or CLI. GitCask validates repository access before the repo is admitted.

02

Dispatch the backup job

Scheduled work or manual triggers create jobs that move through queueing, execution, and completion without requiring an admin UI.

03

Clone, compress, verify

A dedicated container performs the mirrored clone, packages the archive, and computes a checksum before anything is uploaded.

04

Store the artifact and metadata

Backups land in R2 with associated repository details and run records so operators can audit what happened and when.

Automation surface

Operate it from scripts.

The product is intentionally closer to infrastructure than to a dashboard. Most teams will interact with it through curl, cron, CI, or a thin wrapper around the CLI.

  • Per-repository schedules with manual triggers
  • Authenticated admin API for repos, jobs, and runs
  • Retention cleanup and retry behavior built into the worker
CLI / API Minimal control surface 
# register a repository backup target
gitcask repos add acme/api --interval 60

# trigger an immediate run
gitcask repos trigger repo_01
